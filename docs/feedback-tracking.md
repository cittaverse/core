## 反馈分类与优先级

### 分类体系

| 类别 | 说明 | 示例 |
|------|------|------|
| **Bug** | 功能异常、错误 | "脚本运行报错" |
| **体验问题** | 可用性问题 | "文档太难懂" |
| **功能建议** | 新功能请求 | "希望能支持 XX" |
| **文档问题** | 文档错误或缺失 | "XX 步骤没说清楚" |
| **性能问题** | 速度慢、资源占用高 | "运行太慢了" |
| **其他** | 不属于以上类别 | "感谢分享" |

### 优先级定义

| 优先级 | 标准 | 响应时间 | 处理时间 |
|--------|------|----------|----------|
| **P0 - 紧急** | 核心功能不可用、数据丢失 | <4h | <24h |
| **P1 - 高** | 重要功能受限、影响多数用户 | <24h | <3 天 |
| **P2 - 中** | 一般问题、有替代方案 | <48h | <1 周 |
| **P3 - 低** | 优化建议、小问题 | <1 周 | 排期处理 |

## 反馈处理流程

### 标准流程

```
收到反馈 → 分类标记 → 优先级评估 → 分配处理 → 修复/回复 → 关闭 → 沉淀经验
```

### 详细步骤

#### Step 1: 收到反馈

- GitHub Issue: 自动标记 `triage`
- 知乎评论：手动记录到反馈表
- 私信/邮件：手动创建 GitHub Issue

#### Step 2: 分类标记

添加对应标签：
- `bug` / `enhancement` / `documentation` / `question`
- `feedback` / `feature-request`

#### Step 3: 优先级评估

根据影响范围和紧急程度标记：
- `priority: P0` / `priority: P1` / `priority: P2` / `priority: P3`

#### Step 4: 分配处理

- P0/P1: 立即处理，@V 知晓
- P2: 纳入本周迭代计划
- P3: 纳入 backlog，定期评审

#### Step 5: 修复/回复

- Bug: 修复后回复，附 commit 链接
- 问题：详细解答，更新 FAQ
- 建议：评估后回复是否采纳

#### Step 6: 关闭

- 确认问题解决
- 邀请用户验证
- 关闭 Issue

#### Step 7: 沉淀经验

- 高频问题 → 更新 FAQ
- 重要建议 → 纳入路线图
- Bug 根因 → 更新测试用例
